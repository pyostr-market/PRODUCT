# Архитектурные правила

Документ определяет архитектурные принципы разработки приложений.
Эти правила обязательны для соблюдения при создании новых модулей и изменении существующих.
Цель — поддерживать предсказуемую архитектуру, доменно-ориентированную модель и масштабируемость системы.

---

# 1. Общие принципы

1. Архитектура системы строится на основе **Domain-Driven Design (DDD)** и **CQRS**.
2. Бизнес-логика должна находиться в доменной модели.
3. Application слой должен только координировать выполнение операций.
4. Domain слой должен быть независимым от инфраструктуры и фреймворков.
5. Код должен отражать **Ubiquitous Language** домена.
6. Архитектура должна оставаться понятной и предсказуемой для всех модулей системы.

---

# 2. Границы домена (Bounded Context)

Система разделяется на **доменные контексты**.

Контекст — это модуль, который отвечает за конкретную область бизнеса.

Примеры контекстов:

* Users
* Billing
* Orders
* Catalog
* Notifications

Каждый контекст:

* имеет собственную доменную модель
* содержит агрегаты
* содержит application-логику
* может иметь собственные инфраструктурные реализации

Контексты могут зависеть друг от друга через импорт модулей, однако необходимо соблюдать следующие правила:

1. Избегать циклических зависимостей между контекстами.
2. Контексты не должны напрямую изменять агрегаты других контекстов.
3. Взаимодействие между контекстами должно происходить через:

   * интерфейсы
   * сервисы
   * события
   * read-модели.

---

# 3. Структура модуля

Каждый доменный модуль должен иметь следующую структуру:

```
module_name/

    domain/
        entities/
        value_objects/
        aggregates/
        services/
        events/
        repositories/

    application/
        commands/
        command_handlers/
        queries/
        query_handlers/

    infrastructure/
        repositories/
        messaging/
        persistence/

    api/
```

---

# 4. Domain Layer

Domain слой содержит **бизнес-логику системы**.

Domain слой должен включать:

* Entities
* Value Objects
* Aggregates
* Domain Services
* Domain Events
* Repository Interfaces

Domain слой **не должен зависеть от**:

* баз данных
* ORM
* HTTP
* фреймворков
* инфраструктуры

Domain код должен быть чистой бизнес-логикой.

---

# 5. Entities

Entity — объект домена, имеющий **идентичность**.

Entity должна:

* иметь уникальный идентификатор
* содержать бизнес-поведение
* защищать свои инварианты

Нельзя использовать entity как простой DTO.

Плохо:

```
class Order {
    id
    status
}
```

Хорошо:

```
class Order {

    cancel()
    approve()
    addItem()
}
```

Все изменения состояния должны происходить через методы сущности.

---

# 6. Value Objects

Value Object — неизменяемый объект, определяемый значением.

Примеры:

* Money
* Email
* Address
* Quantity
* Price

Value Objects:

* не имеют идентичности
* являются immutable
* используются вместо примитивных типов

Плохо:

```
string email
decimal price
```

Хорошо:

```
Email
Money
Price
```

---

# 7. Aggregates

Aggregate — транзакционная граница домена.

Каждый агрегат должен иметь **Aggregate Root**.

Правила:

1. Изменения выполняются только через aggregate root.
2. Внешний код не должен напрямую изменять внутренние сущности агрегата.
3. Один репозиторий соответствует одному aggregate root.
4. Одна транзакция должна изменять только один агрегат.

Пример:

```
Order (Aggregate Root)
 ├─ OrderItem
 └─ OrderStatus
```

---

# 8. Domain Services

Domain Service используется, если бизнес-логика:

* не принадлежит конкретной сущности
* работает с несколькими агрегатами
* представляет доменную операцию

Domain Service должен содержать **чистую бизнес-логику**.

---

# 9. Domain Events

Domain Event отражает факт изменения в домене.

Примеры:

```
OrderCreated
PaymentCompleted
InvoiceApproved
```

Domain события используются для:

* реакций внутри контекста
* интеграции между контекстами
* обновления read-моделей.

---

# 10. CQRS

Система использует **Command Query Responsibility Segregation**.

### Command

Command представляет намерение изменить состояние.

Примеры:

```
CreateOrder
CancelOrder
ApprovePayment
```

Command:

* не возвращает данные
* инициирует изменение состояния.

---

### Command Handler

Command Handler:

* принимает команду
* загружает агрегат
* вызывает методы агрегата
* сохраняет агрегат через repository.

---

### Query

Query используется только для чтения данных.

Query:

* не изменяет состояние
* возвращает DTO или read-модель.

---

### Query Handler

Query Handler:

* выполняет оптимизированные запросы
* может использовать SQL
* может использовать denormalized read-модели.

---

# 11. Репозитории

Repository — абстракция для работы с агрегатами.

Правила:

1. Repository создается для Aggregate Root.
2. Domain слой содержит только интерфейс репозитория.
3. Реализация находится в infrastructure.

Пример:

```
OrderRepository
InvoiceRepository
```

Запрещено:

```
GenericRepository<T>
```

---

# 12. Application Layer

Application слой отвечает за **оркестрацию use-cases**.

Application слой содержит:

* Commands
* Command Handlers
* Queries
* Query Handlers

Application слой **не должен содержать бизнес-правила**.

---

# 13. Infrastructure Layer

Infrastructure слой содержит технические реализации:

* базы данных
* ORM
* messaging
* external APIs

Infrastructure реализует:

* репозитории
* интеграции
* persistence.

---

# 14. Interface Layer (API)

Interface (API) слой отвечает за взаимодействие с внешним миром.

Примеры:

* HTTP controllers
* API endpoints
* message consumers
* CLI.

Interface (API) слой:

* принимает запрос
* формирует command или query
* передает его в application слой.

---

# 15. Поток выполнения команды

Типичный поток выполнения команды:

```
Controller
   ↓
Command
   ↓
CommandHandler
   ↓
Aggregate
   ↓
DomainEvent
   ↓
Repository Save
```

---

# 16. Поток выполнения запроса

Типичный поток запроса:

```
Controller
   ↓
Query
   ↓
QueryHandler
   ↓
Read Model
   ↓
Response DTO
```

---

# 17. Зависимости слоев

Разрешенные зависимости:

```
interface(API) → application → domain
infrastructure → domain
```

Domain слой не должен зависеть от других слоев.

---

# 18. Запрещенные практики

Запрещено:

* бизнес-логика в контроллерах
* бизнес-логика в query handlers
* использование entity как DTO
* прямое изменение внутренних сущностей агрегата
* циклические зависимости между контекстами
* generic repositories
* использование ORM-аннотаций в domain модели.

---

# 19. Требования к новому коду

При добавлении нового функционала необходимо:

1. определить доменный контекст
2. определить агрегат
3. определить команды
4. определить domain события
5. реализовать command handlers
6. реализовать query handlers
7. реализовать read-модели.

---

# 20. Основной принцип

Архитектура должна отражать **бизнес-домен**, а не технические детали.

Код должен быть организован вокруг **бизнес-операций и моделей**, а не вокруг фреймворков или инфраструктуры.


# 21. Метрики качества архитектуры

Можно ввести оценку по шкале.

Категория	Оценка
Четкость контекстов	0–5
Доменная модель	0–5
Aggregates	0–5
CQRS separation	0–5
Layer isolation	0–5
Dependency direction	0–5


# 22. Выявление архитектурных рисков

После анализа выделяются зоны риска:

Critical

- циклические зависимости контекстов
- сломанные агрегаты
- бизнес-логика в application

High

- anemic domain model
- shared entities между контекстами

Medium

- отсутствие value objects
- слабый CQRS
