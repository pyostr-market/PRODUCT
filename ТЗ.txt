Техническое задание
    CMS-модуль для платформы Marketplace
    1. Общая информация

    CMS-модуль предназначен для управления контентом платформы Marketplace без необходимости изменения кода приложения.
    Модуль должен позволять администраторам управлять информационными страницами, контентными блоками, FAQ, email-шаблонами,
    SEO-данными и системными флагами.

    CMS реализуется как backend-модуль на FastAPI с использованием существующей системы загрузки изображений (Media module).
    Фронтенд (React) получает контент через API и динамически рендерит страницы на основе структуры блоков.

    2. Основные задачи CMS

    CMS должна обеспечивать:

    управление контентными страницами сайта

    редактирование блоков страниц (баннеры, карусели, текстовые блоки и т.п.)

    управление FAQ

    хранение и редактирование email-шаблонов

    управление feature flags (включение/отключение функций)

    базовые SEO настройки страниц

    3. Управление страницами сайта (Pages)

    CMS должна позволять создавать и редактировать информационные страницы сайта.

    Примеры страниц

    Главная страница

    О нас

    Контакты

    Assets

    FAQ

    Политика конфиденциальности

    Пользовательское соглашение

    Основные поля страницы

    ID

    slug (уникальный идентификатор страницы)

    title (название страницы)

    is_published (статус публикации)

    created_at

    updated_at

    Поведение

    страница может быть опубликована или скрыта

    страница содержит набор контентных блоков

    порядок блоков на странице может изменяться

    4. Контентные блоки страниц (Page Blocks)

    Страница состоит из набора блоков.
    Каждый блок представляет отдельный элемент интерфейса.

    Примеры блоков

    carousel (карусель)

    banner (баннер)

    text (текстовый блок)

    image_text (блок изображение + текст)

    hero (главный баннер)

    CTA блок

    FAQ блок

    информационные блоки

    Поля блока

    ID

    page_id

    type (тип блока)

    order (порядок отображения)

    data (JSON-структура данных блока)

    is_active

    created_at

    updated_at

    Принцип работы

    Тип блока определяет React-компонент, который будет его отображать.
    Поле data хранит структуру контента в формате JSON.

    Пример структуры блока

    Карусель:

    {
      "slides": [
        {
          "title": "Big Sale",
          "subtitle": "Up to 50%",
          "image_id": 12,
          "button_text": "Shop now",
          "button_url": "/catalog"
        }
      ]
    }

    Текстовый блок:

    {
      "title": "About our marketplace",
      "content": "<p>HTML content</p>"
    }

    Изображения в блоках хранятся через upload_id, полученный из модуля Media.

    5. FAQ (Часто задаваемые вопросы)

    CMS должна позволять управлять списком часто задаваемых вопросов.

    Поля FAQ

    ID

    question

    answer

    category

    order

    is_active

    created_at

    Возможности

    создание и редактирование вопросов

    сортировка

    группировка по категориям

    включение/отключение отображения

    FAQ могут использоваться как:

    отдельная страница

    блок внутри страницы

    6. Email шаблоны

    CMS должна позволять хранить и редактировать email-шаблоны, используемые системой уведомлений.

    Примеры email-уведомлений

    успешная регистрация

    подтверждение email

    восстановление пароля

    уведомления о действиях с товарами

    системные уведомления

    Поля шаблона

    ID

    key (уникальный идентификатор шаблона)

    subject

    body_html

    body_text

    variables (список используемых переменных)

    is_active

    created_at

    updated_at

    Использование переменных

    Шаблон может содержать переменные:

    {{user_name}}
    {{verify_link}}
    {{order_id}}

    При отправке письма система подставляет реальные значения.

    7. Feature Flags

    Feature Flags позволяют включать или отключать функциональность платформы без изменения кода.

    Примеры

    включение/отключение чата

    разрешение регистрации

    включение личного кабинета

    отображение отдельных страниц

    режим обслуживания

    Поля

    ID

    key

    enabled

    description

    updated_at

    8. SEO настройки

    CMS должна предоставлять базовые SEO настройки для страниц.

    Поля

    ID

    page_slug

    title

    description

    keywords

    og_image_id

    SEO данные используются фронтендом для генерации meta-тегов.

    9. API взаимодействие

    Контент CMS должен быть доступен через REST API.

    Примеры endpoints

    Получение страницы:

    GET /cms/pages/{slug}

    Ответ:

    {
      "page": "home",
      "blocks": [
        {
          "type": "carousel",
          "data": {}
        },
        {
          "type": "text",
          "data": {}
        }
      ]
    }

    Получение FAQ:

    GET /cms/faq

    Получение feature flags:

    GET /cms/feature-flags
    10. Интеграция с фронтендом

    Фронтенд (React) должен:

    запрашивать страницы через API

    динамически рендерить блоки на основе поля type

    передавать данные блока в соответствующий React-компонент


    11. Административный интерфейс

    Административная часть должна позволять:

    создавать и редактировать страницы

    добавлять и удалять блоки

    менять порядок блоков

    редактировать FAQ

    управлять email-шаблонами

    включать и отключать feature flags

    редактировать SEO настройки

    12. Перспективы расширения

    В будущем CMS может быть расширена следующими возможностями:

    версия контента (versioning)

    предпросмотр страниц

    планирование публикации

    мультиязычность

    A/B тестирование контента

    расширенные SEO настройки
